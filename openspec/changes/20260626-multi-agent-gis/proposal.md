# Proposal: 多智能体协同 GIS 应急助手

## 1. Why（背景与问题）

### 1.1 业务背景
京津冀城市综合防灾应急管理系统当前以"按钮式"交互为主：用户需手动打开面板、填写参数、点击执行，才能完成专题检索、空间查询、缓冲区分析、网络分析等操作。对于应急指挥场景，操作链路长、学习成本高，不利于快速决策。

### 1.2 核心问题
| 问题 | 描述 |
|------|------|
| **交互门槛高** | 用户必须记住每个功能在哪个菜单、需要哪些参数，应急场景下效率低下 |
| **功能孤立** | 专题检索、空间分析、网络分析各自独立，无法自动串联完成"灾情评估→资源匹配→路径规划"的完整应急流程 |
| **知识脱节** | 前沿救援方案、应急预案等知识存在于文档中，无法与 GIS 操作实时联动 |
| **缺决策辅助** | 系统只返回数据结果，不提供分析建议和方案优化 |

### 1.3 为什么现在做
- 课程设计需要亮点和创新性，纯 GIS 功能已较完善
- 多智能体 + RAG 是当前 AI 应用的前沿方向
- 现有代码模块化较好（各功能组件独立），适合通过 Agent 编排集成

---

## 2. What（变更内容）

### 2.1 新增能力概述
在现有 GIS 系统基础上，新增**多智能体协同应急助手**，以侧边栏对话面板形式存在，用户通过自然语言即可触发 GIS 操作和应急分析。

### 2.2 核心功能模块

#### 模块一：对话式交互界面
- 页面右侧可展开/收起的聊天面板
- 支持多轮对话、消息历史
- 支持工具调用过程可视化（"思考中→正在执行缓冲区分析→完成"）

#### 模块二：GIS 工具调用（Tool Layer）
将现有 GIS 功能封装为 Agent 可调用的 Tool：
| Tool | 对应功能 | 输入参数 | 输出 |
|------|---------|---------|------|
| `feature_search` | 专题检索 | keyword, level | 要素列表 + 地图标绘 |
| `spatial_query` | 空间查询 | geometry, mode | 要素列表 + 地图标绘 |
| `buffer_analysis` | 缓冲区分析 | geometry, distance | 缓冲区面 + 地图标绘 |
| `overlay_analysis` | 叠置分析 | sourceDataset, operateDataset, operation | 分析结果 + 地图标绘 |
| `shortest_path` | 最短路径 | points (≥2) | 路径线 + 距离 + 地图标绘 |
| `service_area` | 服务区分析 | center, radius | 服务区面 + 地图标绘 |
| `fly_to` | 地图定位 | location | 地图视图变化 |

#### 模块三：RAG 知识库
- 向量数据库存储应急救援方案、应急预案、灾害处置手册等
- Agent 在回答应急相关问题时自动检索知识库补充上下文
- 支持知识库内容管理（简易的文档上传/索引）

#### 模块四：多智能体协同（Multi-Agent）
采用"1 个协调者 + N 个专业 Agent"架构：
| Agent | 职责 | 调用工具 |
|-------|------|---------|
| **Coordinator（协调者）** | 意图理解、任务拆解、结果汇总 | LLM + 所有子 Agent |
| **SearchAgent（检索专家）** | 专题检索、空间查询、定位 | feature_search, spatial_query, fly_to |
| **AnalysisAgent（分析专家）** | 缓冲区、叠置分析、影响评估 | buffer_analysis, overlay_analysis |
| **RouteAgent（路径专家）** | 最短路径、服务区、多车调度 | shortest_path, service_area |
| **KnowledgeAgent（知识专家）** | RAG 检索、方案生成 | rag_retrieval |

**协同流程（应急场景示例）**：
```
用户："朝阳区发生地震，帮我评估灾情并规划救援路线"
  │
  ▼
Coordinator 拆解任务：
  ├─ 任务1：SearchAgent → 定位朝阳区，查询周边设施
  ├─ 任务2：AnalysisAgent → 做缓冲区分析评估影响范围
  ├─ 任务3：KnowledgeAgent → 检索地震救援方案
  └─ 任务4：RouteAgent → 规划从最近医院到震中的最短路径
  │
  ▼
  分步执行，每步结果实时标绘在地图上
  │
  ▼
  Coordinator 汇总，生成综合报告
```

### 2.3 算法层（创新性）
在传统 Tool Calling 基础上，增加确定性算法约束：
- **Pareto 多目标匹配**：资源调度时在"距离、容量、紧急度"间做权衡
- **ACO 蚁群算法（mTSP）**：多点对多点的多车路径规划（超越 iServer 单对单最短路径）

## 3. Capabilities（新增/修改能力）

### 3.1 新增能力
| 能力 | 说明 |
|------|------|
| 自然语言 GIS 操作 | 用对话替代按钮操作，降低使用门槛 |
| 多步骤自动编排 | Agent 自动串联多个 GIS 功能完成复杂任务 |
| RAG 知识增强 | 应急方案、处置流程等知识可被调用 |
| 多智能体协同 | 专业分工 + 结构化调度，比单 Agent 更可靠 |
| 分步可视化 | 每步分析结果实时标绘到地图，过程透明 |

### 3.2 修改能力
| 模块 | 变更说明 |
|------|---------|
| Pinia Store | 新增 agent 相关状态（对话历史、当前工具执行状态） |
| 地图组件 | 暴露可被 Agent 调用的 API（通过 store 或事件总线） |
| 各功能组件 | 部分逻辑抽离为可复用的工具函数，供 Agent 直接调用 |

## 4. Impact（影响范围）

### 4.1 前端影响
| 目录/文件 | 影响程度 | 说明 |
|-----------|---------|------|
| `src/components/` | 新增 | 新增 AgentChat 聊天面板组件 |
| `src/stores/` | 修改 | 新增 agent store，扩展 map store |
| `src/utils/agent/` | 新增 | Agent 核心逻辑（tools、agents、rag、coordinator） |
| `src/views/` | 轻微修改 | 在主页挂载聊天面板 |
| `App.vue` | 轻微修改 | 引入聊天面板 |

### 4.2 后端影响
| 模块 | 影响程度 | 说明 |
|------|---------|------|
| 后端整体 | 可选 | RAG 向量库可前端纯 JS 实现，也可后端提供 API |

### 4.3 第三方依赖
| 依赖 | 用途 | 必需性 |
|------|------|--------|
| `@langchain/core` | Agent 框架基础 | 可选（也可自实现） |
| 向量库（如 `vectra` 或 `hnswlib`） | RAG 向量检索 | 必需 |
| LLM API | 大语言模型调用 | 必需 |

### 4.4 非功能影响
- **性能**：LLM 调用有网络延迟（1-5s），需做加载状态提示
- **安全**：前端不存 API Key，通过后端代理或 Vite dev proxy
- **兼容性**：仅现代浏览器，需 WebAssembly 支持（如果向量库用 WASM）

## 5. 难点与关键决策点

### 5.1 核心难点：Agent 如何操作 GIS 功能？
**问题**：现有的 FeatureSearch、SpatialAnalysis 等组件都是 UI 驱动的，Agent 怎么调用？

**初步思路**（design 阶段详细论证）：
1. **方案 A：组件逻辑抽离到 composables** — 将业务逻辑（搜索、分析函数）从组件中抽离到 `utils/` 或 `composables/`，组件和 Agent 都调用这些纯函数
2. **方案 B：事件总线 / Store Action** — Agent 通过触发事件或调用 store action 来操作，组件监听状态变化
3. **方案 C：直接操作 map 实例** — Agent 通过 store 拿到 map 实例，自己执行 addSource/addLayer

初步倾向：**方案 A + B 结合**，核心 GIS 操作封装为纯函数（可测试、可复用），通过 store 暴露给 Agent 和组件共享。

### 5.2 其他难点
- 多智能体调度的状态管理和收敛控制
- RAG 知识库的数据来源和索引构建
- LLM 调用的成本和延迟控制
