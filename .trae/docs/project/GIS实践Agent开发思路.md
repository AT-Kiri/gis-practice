# AI/Agent 创新方案设计报告
## 京津冀城市综合防灾应急管理系统
## 1. 问题分析
| 方案 | 本质 | 问题 |
|------|------|------|
| 纯 AI 对话 | LLM 文本生成 | 只输出文字，不产生实际 GIS 操作 |
| LLM + Tool | 自然语言替代按钮点击 | 交互方式变了，操作能力没变 |
| + RAG | 向量库补充上下文 | 本项目数据范围固定，关系数据库也能实现 |
核心结论：AI 不能只做"理解意图 → 调接口 → 包装结果"的简单链路，必须参与更深层的决策优化。

## 2. 方案：三层架构
┌─────────────────────
│          智能体层 (Agent)            │
│  Coordinator → 子Agent 协同调度     │
│  (LLM 负责意图理解与任务拆解)         │
├──────────────────────
│          算法层 (Algorithm)          │
│  Pareto 多目标匹配 + ACO 路径优化    │
│  (确定性算法，替代 LLM 不可靠的推理)   │
├──────────────────────
│          工具层 (Tool)               │
│  包装 iServer REST API              │
│  (已有部分，可以扩展)                 │
└──────────────────────
### 2.1 工具层（已有部分，可以扩展）
将现有或将来的 GIS 操作转化为标准 Tool：buffer_analysis、spatial_query、shortest_path 等。
### 2.2 算法层（核心创新）
LLM 不擅长数学计算，因此把优化任务剥离给算法模块：

Pareto 多目标匹配：在距离、容量、紧急度之间权衡，输出一组最优解（非单一答案）
ACO 蚁群算法 (mTSP)：多点对多点的多车路径规划，iServer 只算了单对单最短路径

### 2.3 智能体层（核心创新）
| Agent | 职责 | 使用工具/算法 |
|-------|------|---------------|
| Coordinator | 意图理解、拆任务、汇总 | LLM |
| 灾情评估 | 评估影响范围与设施 | buffer, query |
| 资源匹配 | 供需匹配优化 | query, Pareto |
| 路径优化 | 多车配送路径规划 | path, ACO |
| 方案综合 | 生成报告、触发可视化 | LLM |

## 3. 设计原则：
Agent 间通过固定 JSON Schema（或其他格式） 通信，不自由理解和发挥
调度逻辑由代码控制（next.agent 只能取预定义枚举值）
不收敛则循环执行，上限 N 次

真 Multi-Agent：结构化契约 + 代码级调度 + 收敛循环，不是共享上下文调多个 Tool
分步可视化：每步结果实时（或非实时但按顺序）标绘在地图上（受灾范围 → 物资点 → 路径）



## 4. 原始思考文本：

主题：京津冀城市综合防灾应急管理系统

创新方向：结合AI/Agent



思路:
1、LLM，做方案制定，内容输出的ai对话助手
2、LLM+tool，做Agent，让ai调用tool实现，通过对话的形式来执行空间查询，空间分析和网络分析等操作，
3、+RAG，提高agent的执行效率，来规范或者说避免ai触碰到现有服务和数据的边界。


问题：
1、纯ai对话意义不大，就算在系统提示词层面做专项优化，也只是无效文本输出工具。
2、只是用自然语言替代了按钮点击，没有质变
3、关系数据库也能做，用向量数据库只能算是多用一些技术栈，rag的作用主要是在提示词层面来给ai补充准确及时的信息，但是目前课设对这些要求不是很高。（不过还是要有比较好）

还能做什么，算法融入到哪里？
多智能体（Multi Agent），协同工作，智能体调度可以做算法，专门做某个模块的智能体也能融入算法，比如某个模块主要是接收数据的然后做规划和方案输出的，这块我们不能让ai自由发挥，得用成熟的算法去约束或者说接管那些规划和方案的输出。

而智能体调度部分，不是写死的，也不能让ai自由发挥来决定走不走下一个agent以及如果走下一步走哪个agent，要规范ai的输出格式（避免信息在传递的过程中被曲解含义，以及明确下一步的操作），而且要一定程度上强制性的进行一定轮次的循环执行来确保比较好的最终输出结果。
