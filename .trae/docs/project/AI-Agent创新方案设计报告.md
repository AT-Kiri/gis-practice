# AI/Agent 创新方案设计报告

> 京津冀城市综合防灾应急管理系统

---

## 1. 问题分析

| 方案 | 本质 | 问题 |
|------|------|------|
| 纯 AI 对话 | LLM 文本生成 | 只输出文字，不产生实际 GIS 操作 |
| LLM + Tool | 自然语言替代按钮点击 | 交互方式变了，操作能力没变 |
| + RAG | 向量库补充上下文 | 本项目数据范围固定，关系数据库也能实现 |

**核心结论**：AI 不能只做"理解意图 → 调接口 → 包装结果"的简单链路，必须参与更深层的**决策优化**。

---

## 2. 方案：三层架构

```
┌────────────────────────────────────┐
│          智能体层 (Agent)            │
│  Coordinator → 子Agent 协同调度      │
│  (LLM 负责意图理解与任务拆解)         │
├────────────────────────────────────┤
│          算法层 (Algorithm)          │
│  Pareto 多目标匹配 + ACO 路径优化    │
│  (确定性算法，替代 LLM 不可靠的推理)   │
├────────────────────────────────────┤
│          工具层 (Tool)               │
│  包装 iServer REST API              │
│  (已有代码，复用不改)                 │
└────────────────────────────────────┘
```

### 2.1 工具层（已有，不动）
将现有 GIS 操作为标准 Tool：`buffer_analysis`、`spatial_query`、`shortest_path` 等。

### 2.2 算法层（核心创新）
**LLM 不擅长数学计算，因此把优化任务剥离给算法模块**：

- **Pareto 多目标匹配**：在距离、容量、紧急度之间权衡，输出一组最优解（非单一答案）
- **ACO 蚁群算法 (mTSP)**：多点对多点的多车路径规划，iServer 只算了单对单最短路径

### 2.3 智能体层（核心创新）

| Agent | 职责 | 使用工具/算法 |
|-------|------|-------------|
| Coordinator | 意图理解、拆任务、汇总 | LLM |
| 灾情评估 | 评估影响范围与设施 | buffer, query |
| 资源匹配 | 供需匹配优化 | query, **Pareto** |
| 路径优化 | 多车配送路径规划 | path, **ACO** |
| 方案综合 | 生成报告、触发可视化 | LLM |

**设计原则**：
- Agent 间通过固定 JSON Schema 通信，不自由发挥
- 调度逻辑由代码控制（`next.agent` 只能取预定义枚举值）
- 不收敛则循环执行，上限 N 次

---

## 3. 创新点

1. **三层解耦**：工具（调接口）→ 算法（数学优化）→ 智能体（LLM 决策），各司其职
2. **算法约束 LLM**：LLM 只做理解和解释，数学计算交给确定性算法
3. **真 Multi-Agent**：结构化契约 + 代码级调度 + 收敛循环，不是共享上下文调多个 Tool
4. **分步可视化**：每步结果实时标绘在地图上（受灾范围 → 物资点 → 路径）

## 4. 技术选型

| 模块 | 选型 |
|------|------|
| LLM | DeepSeek API（FP, ¥1/百万 token） |
| Agent 框架 | 轻量自实现（不依赖 LangChain） |
| 算法 | Java（SpringBoot 项目内部） |
| 前端 | AIChatPanel.vue（新组件） |
| 通信 | SSE 流式输出 |
