# AI 编码行为准则

> 本文件规范 AI Agent 在 pad-market 项目中的编码行为原则。
> 源自 [Andrej Karpathy 的 CLAUDE.md](https://github.com/multica-ai/andrej-karpathy-skills/blob/main/CLAUDE.md)，经项目化适配。
>
> **核心权衡**：这些准则偏向谨慎而非速度。对于简单任务，请自行判断。

---

## 1. 编码前思考（Think Before Coding）

**不要假设。不要隐藏困惑。暴露权衡。**

在实现前：

- **明确陈述假设**。如果不确定，主动询问。
- **如果存在多种解释，请呈现出来**——不要默默选择其一。
- **如果存在更简单的方案，请说明**。在适当时候提出反对意见。
- **如果某些地方不清楚，停下来**。指出困惑之处，然后询问。

> 与项目现有规范的衔接：编码前的思考应结合 [project_rules.md](project_rules.md) 中的功能开关自检（§6.4.1）和模块修改权限（§8）进行约束检查。

---

## 2. 简洁优先（Simplicity First）

**最小代码解决问题。不做推测性设计。**

- 不添加超出需求的功能。
- 不为一次性代码做抽象。
- 不添加未被要求的"灵活性"或"可配置性"。
- 不处理不可能发生的场景的错误。
- 如果你写了 200 行而可以用 50 行完成，重写它。

**自检问题**："资深工程师会觉得这过度设计了吗？"如果是，简化。

> 与项目现有规范的衔接：简洁优先原则应服从 [project_rules.md](project_rules.md) 的模块修改权限（§8.1），禁止以"简化"为由修改 `pad-common`、`pad-marketing-service` 等禁止模块。

---

## 3. 精准修改（Surgical Changes）

**只触碰必须修改的。只清理自己制造的。**

编辑现有代码时：

- **不要"改进"相邻的代码、注释或格式**。
- **不要重构未损坏的东西**。
- **匹配现有风格**，即使你自己会采用不同的写法。
- **如果注意到无关的死代码，提及它——但不要删除它**。

当你的变更产生孤儿代码时：

- **移除因你的变更而变得未使用的 import / 变量 / 函数**。
- **不要移除预先存在的死代码**，除非被要求。

**检验标准**：每一行变更都应能直接追溯到用户的请求。

> 与项目现有规范的衔接：精准修改原则与 [project_rules.md](project_rules.md) 的代码审查规范（§10.1）一致——审查时必须定位到实际运行的子类，禁止仅通过父类推断。

---

## 4. 目标驱动执行（Goal-Driven Execution）

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：

| 模糊请求 | 转化为可验证目标 |
|---------|----------------|
| "添加校验" | "为无效输入编写测试，然后让它们通过" |
| "修复 Bug" | "编写一个能复现它的测试，然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

对于多步骤任务，简要陈述计划：

```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

**强成功标准**允许独立循环。**弱标准**（"让它跑起来"）需要不断澄清。

> 与项目现有规范的衔接：目标驱动执行应与 [workflows.md](../docs/workflows.md) 的 6 Phase 协作流程结合，每个 Phase 都有明确的输入、输出和验证标准。

---

## 5. 与项目规范的优先级

当本准则与 [project_rules.md](project_rules.md) 或 [workflows.md](../docs/workflows.md) 冲突时：

| 场景 | 优先级 |
|------|--------|
| 模块修改权限（§8） | **项目规范优先** —— 无论多简洁，禁止修改 `pad-common` 等模块 |
| 功能开关强制约束（§6.4.1） | **项目规范优先** —— 新功能必须配置开关 |
| 安全规范（§7） | **项目规范优先** —— 严禁提交密钥、密码 |
| 代码风格 | **项目规范优先** —— 匹配现有风格，即使本准则建议不同 |
| 实现方式选择 | **本准则优先** —— 在允许范围内选择最简洁方案 |
| 是否询问用户 | **本准则优先** —— 不确定时主动询问，不要假设 |

---

## 6. 效果验证

这些准则生效的标志：

- Diff 中不必要的变更更少
- 因过度设计导致的重写更少
- 澄清性问题出现在实现之前，而不是犯错之后
