# req-doc-gen — 基于 GIS 功能模块的需求文档生成

## 定位

基于项目已有的 GIS 功能模块，结合用户的新需求描述，自动生成规范的二次开发需求文档（requirement.md）。

> **注意**：本技能仅生成需求文档。需求确认后，再根据项目流程生成技术设计文档。

## 触发条件

用户提到以下任何一种表述时触发：
- "帮我基于 [某某] 功能做一个新需求"
- "在 [某某] 功能基础上，增加/修改 [某某] 功能"
- "帮我写一个需求文档"
- "基于现有 GIS 功能，二次开发 [某某] 功能"
- "帮我分析一下这个需求怎么实现"

## 执行步骤

### Step 1: 解析用户需求

从用户输入中提取：
1. **目标功能**：基于哪个现有 GIS 功能进行二次开发（如空间查询、网络分析等）
2. **需求类型**：新增功能 / 功能优化 / Bug修复 / 流程改造
3. **核心变更点**：用户想要修改或新增的具体内容
4. **影响范围**：前端/后端/服务层/数据层

### Step 2: 读取项目概览

从 `project-brief.md` 读取项目整体情况：
1. 读取技术栈和架构
2. 读取现有功能模块清单
3. 读取数据资源（SuperMap iServer 服务）
4. 读取关键业务规则

### Step 3: 分析变更影响

对比现有功能和用户需求，分析：
1. **新增功能**：需要新增哪些功能模块
2. **修改功能**：需要修改哪些现有功能
3. **删除功能**：需要删除哪些功能（如有）
4. **依赖关系**：新功能依赖哪些现有组件/服务/数据

### Step 4: 生成需求文档

生成规范的需求文档，保存到 `.trae/req-docs/{需求ID}/requirement.md`：

**文档结构**：
```markdown
# {需求名称}

## 一、需求背景
[基于项目背景的说明]

## 二、现有功能分析
[引用项目现有功能模块的关键内容]

## 三、需求描述
[用户的具体需求]

## 四、变更范围
### 4.1 前端变更
[前端需要修改的内容]

### 4.2 后端变更
[后端需要修改的内容]

### 4.3 服务层变更
[SuperMap iServer 服务需要修改的内容]

### 4.4 数据变更
[数据层需要修改的内容]

## 五、详细设计
### 5.1 功能流程
[基于现有流程的变更后流程图]

### 5.2 接口设计
[新增/修改的接口定义]

### 5.3 数据设计
[新增/修改的数据结构]

### 5.4 关键逻辑
[新增/修改的业务逻辑]

## 六、验收标准
[可量化的验收条件]

## 七、参考文档
[引用的项目文档]
```

## 需求ID规范

- 格式：`{date}-{short-desc}`，例如：`20260616-flood-simulation-enhance`
- 需求ID全局唯一

## 输出文件规范

| 文件 | 路径 | 格式 | 说明 |
|------|------|------|------|
| 需求文档 | `.trae/req-docs/{需求ID}/requirement.md` | Markdown | 完整需求文档 |

## 使用示例

### 示例1：基于网络分析增加新功能

用户输入："基于网络分析功能，我想增加一个多路径对比功能，可以同时计算并显示多条路径"

执行过程：
1. 解析需求：基于 network-analysis 功能，新增多路径对比
2. 读取 `project-brief.md` 了解现有网络分析功能
3. 分析变更：新增路径列表管理、多路径渲染、路径对比面板
4. 生成文档：`.trae/req-docs/20260616-multi-path-compare/requirement.md`

### 示例2：基于空间查询优化流程

用户输入："空间查询功能我希望可以保存常用的查询条件，下次直接使用"

执行过程：
1. 解析需求：基于 spatial-query 功能，功能优化
2. 读取 `project-brief.md` 了解现有空间查询功能
3. 分析变更：新增查询条件保存/加载、历史记录管理
4. 生成文档：`.trae/req-docs/20260616-query-preset/requirement.md`
